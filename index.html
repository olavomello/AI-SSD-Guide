PART I: FOUNDATIONS

Chapter 1: What Is Specification-Driven Development?

1.1 The Problem With How We Build Software

Let's start with a story every developer knows.

A product manager sends a Slack message: "Can you add a feature to let users export their data?"

You nod, open your IDE, and start coding. Three days later, you deliver a CSV export feature. The product manager looks at it and says: "Oh — I meant PDF. And it should only export the last 30 days. And it needs to include the user's profile photo. And it should email the file, not download it."

You have just experienced the fundamental problem of software development: the gap between what someone means and what someone builds.

This gap is not a people problem. It is not a communication problem. It is not even a technical problem. It is a specification problem.

A specification is a precise, unambiguous description of what a system must do, how it must behave, what constraints it must satisfy, and what outcomes it must produce.

When specifications are missing, implicit, or incomplete, software breaks down — not at runtime, but at the moment someone decided to start coding without thinking clearly about what they were building.

Specification-Driven Development (SDD) is the practice of defining software precisely before building it — and using those definitions as the authoritative source of truth for every person and every tool that participates in the development process.

1.2 A Working Definition

Specification-Driven Development (SDD) is a software engineering methodology in which precise, structured, machine-readable specifications serve as the primary source of truth for design, implementation, testing, review, deployment, and evolution of software systems.

Let's unpack that definition word by word.

Precise — specifications eliminate ambiguity. They say exactly what is required, not approximately.

Structured — specifications follow consistent formats that both humans and machines can parse and reason about.

Machine-readable — specifications can be consumed by AI agents, code generators, test runners, and documentation systems without manual interpretation.

Primary source of truth — specifications are not documentation written after the fact. They are the first artifact produced, and everything else derives from them.

Every person and every tool — specifications are not just for developers. They guide designers, QA engineers, product managers, DevOps engineers, and increasingly, AI coding assistants.

1.3 SDD Is Not New

Specification-driven thinking has existed since the earliest days of computing.

  • 1968: NATO Software Engineering Conference identifies the "software crisis" — the gap between what software should do and what it actually does
  • 1970s: Formal specification languages like Z notation and VDM emerge in academic research
  • 1980s: Interface Definition Languages (IDL) formalize API contracts between systems
  • 1990s: UML brings visual specification to mainstream engineering
  • 2000s: OpenAPI/Swagger standardizes REST API specifications; BDD tools like Cucumber operationalize behavioral specifications
  • 2010s: Infrastructure-as-Code (Terraform, CloudFormation) treats infrastructure as specification
  • 2020s: AI coding assistants make specifications the primary interface between humans and code-generating machines

What has changed is not the concept of specifications, but their reach and impact. In the AI era, a specification is no longer just a document that guides human developers — it is a program that drives machine developers.

1.4 The Four Pillars of SDD

SDD rests on four interlocking principles:

Pillar 1: Specification Before Implementation

No code is written until a specification exists. This is the hardest discipline to maintain and the most valuable when you do. It forces clarity before commitment.

Pillar 2: Specifications Are Living Documents

A specification is not a waterfall artifact that gets filed and forgotten. It evolves with the system. When requirements change, the specification changes first, and the implementation follows the specification.

Pillar 3: Specifications Drive Verification

Tests, reviews, and acceptance criteria derive directly from specifications. A feature is "done" when it satisfies its specification — not when the developer feels good about the code.

Pillar 4: Specifications Enable Automation

Well-structured specifications can be consumed by automated tools — code generators, test generators, documentation generators, and AI agents — to dramatically accelerate development while maintaining consistency.

1.5 What SDD Is Not

Before going further, let's clear up some common misconceptions.

SDD is not waterfall. Waterfall requires complete specifications before any development begins. SDD requires specifications before each piece of development begins — but specifications themselves are iterative and evolve as understanding improves.

SDD is not big design up front (BDUF). BDUF tries to specify everything before building anything. SDD specifies just enough to start building, then adds specification detail as the system grows.

SDD is not documentation-driven development. Documentation describes what was built. Specification prescribes what will be built. SDD treats specification as a first-class engineering artifact, not an administrative task.

SDD is not about bureaucracy. Many developers hear "specification" and think "painful process with lots of approvals." SDD at its best is lightweight, practical, and designed to make developers faster, not slower.

1.6 The SDD Workflow

At its heart, an SDD workflow looks like this:

flowchart TD
    A[Business Requirement] --> B[Business Specification]
    B --> C[Technical Specification]
    C --> D[Architecture Specification]
    D --> E[API Specification]
    E --> F[Database Specification]
    F --> G[Implementation]
    G --> H[Tests derived from Specs]
    H --> I[Review against Specs]
    I --> J[Deployment]
    J --> K[Spec Updated if Behavior Changes]
    K --> B

Notice that specifications flow through every stage. Implementation is just one step in a specification-driven process — not the first step, and not the most important one.

1.7 SDD in the AI Era

Here is where SDD becomes genuinely transformative.

When you work with an AI coding assistant — whether Claude Code, Cursor, GitHub Copilot, or any other tool — you are essentially working with a very capable developer who:

  1. Has no memory between sessions unless you give it one
  2. Has no knowledge of your codebase unless you provide it
  3. Has no understanding of your business rules unless you specify them
  4. Will make assumptions when information is missing, and those assumptions may be wrong
  5. Cannot ask your product manager what they really meant

This means that every gap in your specification becomes a potential hallucination in your AI-generated code.

SDD is the solution. When your specifications are complete, precise, and machine-readable, you give AI agents the context they need to generate code that actually matches what you need.

💡 AI Insight: The quality of AI-generated code is directly proportional to the quality of the specifications provided. Vague specifications produce vague code. Precise specifications produce precise code. This is not a limitation of AI — it is a fundamental property of the relationship between specification and implementation.

### Actors
- **Registered User**: A user with an existing account and verified email address
- **System**: The authentication service

### Preconditions
- The user's email address is verified in the system
- The user's account is active (not suspended or deleted)

### Flow: Happy Path
1. User navigates to /auth/forgot-password
2. User submits their registered email address
3. System validates that email exists and account is active
4. System generates a cryptographically secure 32-byte reset token
5. System stores token hash (SHA-256) with 1-hour expiry in the database
6. System sends reset email with link: /auth/reset-password?token={raw_token}
7. User clicks link within 1 hour
8. System validates token (hash match, not expired, not already used)
9. User submits new password
10. System validates password meets complexity requirements
11. System updates password hash (bcrypt, cost factor 12)
12. System invalidates ALL existing sessions for this user
13. System marks token as used
14. System redirects user to login page with success message

Chapter 2: The History and Evolution of Specifications

2.1 The Software Crisis and the Birth of Specification Thinking

In August 1968, NATO sponsored a conference in Garmisch, Germany, that would change the history of computing. Assembled were fifty of the world's leading computer scientists, brought together to discuss a problem that had grown serious enough to have a name: the software crisis.

The symptoms were alarming. Major software projects were routinely delivered late, over budget, and with dramatically fewer features than planned. When software did ship, it often failed in the field, sometimes catastrophically. The IBM OS/360 project — the largest software project in history at the time — was famously described by its manager, Fred Brooks, as a "tar pit" that consumed teams whole.

The diagnosis that emerged from Garmisch was clear: software development lacked the disciplined engineering practices that other fields took for granted. Civil engineers did not start pouring concrete before drawing blueprints. Electrical engineers did not start soldering components before completing circuit diagrams. But software developers regularly started writing code before anyone had written down what the code was supposed to do.

The prescription was specification: precise, formal descriptions of system behavior that could be reasoned about, verified, and used as the basis for systematic construction.

2.2 Formal Methods (1970s–1980s)

The academic response to the software crisis was formal methods — mathematical techniques for specifying and verifying software behavior.

Z notation (developed at Oxford in the late 1970s) used set theory and predicate logic to describe system state and operations with mathematical precision. A Z specification for a password database might formally prove that passwords could never be retrieved in plaintext form.

VDM (Vienna Development Method) emerged from IBM's Vienna lab and formalized the semantics of programming languages, giving developers a rigorous way to specify what programs meant.

Temporal Logic enabled reasoning about systems that evolve over time — crucial for concurrent and distributed systems.

These approaches were powerful but expensive. Writing formal specifications required significant mathematical expertise and took considerably longer than writing code. They found adoption in safety-critical domains — aerospace, nuclear control systems, medical devices — where the cost of failure justified the cost of formalism.

The lesson of formal methods is important: the more precise a specification, the more it can be reasoned about and verified. We will revisit this principle repeatedly as we discuss AI-driven development.

2.3 Interface Definition Languages (1980s–1990s)

As software systems grew larger and teams grew more distributed, a practical specification need emerged: how do components talk to each other?

Interface Definition Languages (IDLs) answered this question. Rather than specifying entire systems formally, IDLs specified the boundaries between components — the contracts that had to be honored for integration to work.

CORBA IDL (1991) defined distributed object interfaces across language and platform boundaries. A specification in CORBA IDL could be automatically compiled into client and server code in C, C++, Java, or COBOL.

This was a pivotal moment: specifications were becoming executable. You wrote a specification, and tools generated code from it. Sound familiar?

The pattern of "specify the interface, generate the implementation scaffolding" would recur throughout the history of software engineering and is now the dominant paradigm in AI-assisted development.

2.4 UML and Visual Specifications (1990s–2000s)

The Unified Modeling Language (UML), standardized by the Object Management Group in 1997, brought specification to the mainstream by making it visual.

UML gave developers: - Class diagrams to specify data structures and their relationships - Sequence diagrams to specify interaction flows between components - State diagrams to specify system behavior over time - Use case diagrams to specify what the system does from a user perspective - Component diagrams to specify system architecture

For a brief period in the early 2000s, "model-driven development" promised that visual specifications would become the primary programming medium — tools like IBM Rational Rose would generate entire applications from UML models.

This promise was never fully realized. UML models had a fatal flaw: they were hard to keep synchronized with code. The moment a developer changed the implementation without updating the diagram, the specification became a lie.

This lesson — specifications that cannot be automatically verified against implementations will drift — is one of the most important insights in software engineering history. We will address it directly in Chapter 8 when we discuss repository engineering.

2.5 API Specifications and OpenAPI (2010s)

The explosion of web APIs in the 2010s created a practical specification need at massive scale. When Stripe, Twilio, Salesforce, and thousands of other companies were publishing APIs for external developers to consume, the API contract became a product in itself.

Swagger (later renamed OpenAPI Specification) emerged as the standard. An OpenAPI specification describes: - Every endpoint's URL and HTTP method - Every request parameter and its type - Every request body schema - Every response code and its body schema - Authentication requirements - Rate limiting information

OpenAPI specifications are machine-readable YAML or JSON. From a single specification file, tools can: - Generate client SDKs in any programming language - Generate server stub code - Generate interactive API documentation - Run automated API contract tests - Mock the API for frontend development without a real backend

This is SDD in its most mature practical form: a single specification file that drives code generation, documentation, testing, and tooling simultaneously.

OpenAPI's success demonstrated a crucial principle: when specifications are machine-readable and tool-supported, adoption follows. Developers who never thought of themselves as specification-driven adopted OpenAPI because the tooling made it the path of least resistance.

2.6 Infrastructure as Code (2010s)

While OpenAPI was standardizing API specifications, a parallel revolution was happening in operations: Infrastructure as Code (IaC).

The insight was simple and profound: your servers, networks, databases, and load balancers are just another kind of specification. If you write them down precisely in a machine-readable format, you can: - Reproduce identical environments reliably - Version control your infrastructure - Review infrastructure changes before applying them - Automatically verify that infrastructure matches its specification

Tools like Terraform, AWS CloudFormation, Ansible, and Kubernetes manifests all embody specification-driven thinking for infrastructure.

A Terraform configuration does not describe how to create infrastructure — it describes what infrastructure should exist. The tool figures out the how. This is a critical distinction: declarative specifications describe desired state; imperative programs describe process. Declarative wins for large-scale system management because desired state is easier to reason about, compose, and verify than process.

2.7 BDD and Executable Specifications (2010s)

Behavior-Driven Development (BDD), popularized by Dan North and tools like Cucumber, represented another milestone: specifications that are simultaneously human-readable business requirements and machine-executable tests.

Feature: Password Reset

  Scenario: Successful password reset
    Given a registered user with email "user@example.com"
    When the user requests a password reset
    And the user receives a reset email
    And the user clicks the reset link within 1 hour
    And the user submits a new valid password
    Then the user's password is updated
    And all existing sessions are invalidated
    And the user is redirected to the login page

  Scenario: Expired reset link
    Given a registered user with email "user@example.com"
    When the user requests a password reset
    And 61 minutes pass
    And the user clicks the reset link
    Then the user sees an "Link expired" message
    And the user is offered the option to request a new reset link

BDD Gherkin syntax is simultaneously: - A business specification written in plain English - A machine-executable test that verifies the implementation - Documentation that stays synchronized with behavior (because it is the behavior)

BDD solved the specification drift problem that plagued UML: because the specification is the test, if the code doesn't match the specification, the CI pipeline turns red. There is no way for specification and implementation to diverge silently.

2.8 The AI Revolution (2020s–Present)

Everything in the previous sections built toward the moment we are in now.

AI coding assistants — Claude Code, Cursor, GitHub Copilot, Codeium, Amazon Q, Gemini CLI, and a dozen others — are transforming software development at a speed that no previous tool has matched.

These systems are specification-driven at their core. When you prompt Claude Code to "build a REST API for user authentication with JWT tokens, refresh token rotation, and rate limiting", you are writing a specification. The AI interprets the specification and generates an implementation.

But here is the critical insight that this chapter of history teaches us: every specification principle we have learned over 50 years applies to AI prompting.

  • Precision matters: Vague specifications produce wrong implementations (1970s formal methods lesson)
  • Interface contracts must be explicit: Define API shapes before asking for implementation (IDL lesson)
  • Executable specifications prevent drift: Use BDD-style acceptance criteria to verify AI output (BDD lesson)
  • Declarative beats imperative: Tell AI what you want, not how to build it (IaC lesson)
  • Machine-readable wins: Structured specifications in consistent formats work better than prose paragraphs (OpenAPI lesson)

The history of software specification is the history of the field learning, slowly and painfully, how to communicate precisely with machines. AI coding assistants are just the latest — and most powerful — machines we need to communicate with.

2.10 What History Teaches Us

Looking at this arc, several principles emerge that we will apply throughout this guide:

  1. Specification always wins over intuition — every major improvement in software quality has involved making implicit knowledge explicit through specification

  2. Machine-readable specifications unlock automation — the step from "document for humans" to "document for machines" is where exponential value gets created

  3. Specifications must stay synchronized — specifications that can drift from implementations are harmful; synchronization must be automated

  4. The right level of formalism matters — too formal and no one will write or read specifications; too informal and machines cannot process them

  5. Tools drive adoption — specifications become standard practice when tooling makes them the easy path, not the hard path

These principles are the foundation on which we will build our SDD practice in the chapters that follow.

Chapter 3: Why AI Makes SDD Essential

3.1 The Fundamental Nature of AI Coding Assistants

To understand why SDD is essential in the AI era, you first need to understand something important about how AI coding assistants actually work.

When you ask Claude Code to write a function, it does not look at your codebase, understand your business domain, remember your architectural decisions, and carefully craft code that fits your exact context.

What it actually does is: given everything in its current context window, it generates the statistically most likely sequence of tokens that represents good code for the specification it has been given.

This is not a criticism — it is the most important technical fact for SDD practitioners to internalize. It means:

  • Context determines quality
  • What is in the context window determines what the AI knows
  • What is not in the context window, the AI must guess
  • What the AI guesses, it will guess based on patterns in training data — which may or may not match your specific requirements

Every hallucination, every wrong assumption, every misaligned implementation from an AI coding assistant can be traced back to a gap in the specification or context provided.

SDD closes those gaps.

3.2 The Three Failure Modes of AI Development Without SDD

When developers use AI coding assistants without specification discipline, three failure modes reliably emerge.

Failure Mode 1: The Wrong Feature Problem

What happens: The developer asks for a feature. The AI builds something. The developer accepts it. The something is technically functional but not what was needed.

Example: "Add authentication to the API." The AI adds JWT authentication with tokens that never expire and no refresh mechanism. The developer ships it. Three months later, security team flags it.

Root cause: "Authentication" is not a specification. It is a category. The AI filled in the details with training data defaults — which may not match your security requirements.

SDD solution: The specification defines token expiry (15 minutes access, 7 days refresh), refresh token rotation, storage requirements (httpOnly cookies, not localStorage), and revocation mechanism.

Failure Mode 2: The Context Erosion Problem

What happens: A developer uses an AI assistant for days or weeks on a project. Early sessions produce good code. Later sessions produce code that contradicts earlier decisions because the AI has no memory of them.

Example: In session 1, the architect decides to use repository pattern for data access. In session 8, the developer asks for a new feature. The AI generates code that queries the database directly because it has no knowledge of the architectural decision made seven sessions ago.

Root cause: AI assistants have no persistent memory between sessions. Every session starts fresh. Without a mechanism to restore context, decisions made previously are unknown to the current session.

SDD solution: Architectural decisions are captured in specification files (ARCHITECTURE.md, AGENTS.md, ADR files) that are automatically loaded into every AI session, ensuring consistency.

Failure Mode 3: The Specification Drift Problem

What happens: The AI generates code. The developer modifies it. The modified code no longer matches the original specification. Future AI sessions are given the specification, generate code that contradicts the existing implementation, and the codebase accumulates inconsistencies.

Example: The specification says the API returns paginated results with { data: [], pagination: { page, limit, total } }. The developer modified the AI-generated code to use a different shape. Six months later, a new developer asks AI to generate a frontend that consumes this API. The AI generates frontend code based on the specification. The frontend and backend don't match.

Root cause: Specifications and implementations diverge because there is no automated mechanism to keep them synchronized.

SDD solution: Specifications are stored in the repository alongside code. Changes to implementations require corresponding specification updates. Automated tests verify that implementations match specifications.

3.4 The Context Window Is the Specification Interface

The context window is the most important concept in AI-assisted development. It is the bounded region of text that the AI model can "see" at any given moment. Everything inside the context window influences the AI's output. Everything outside it is invisible.

Context windows have grown dramatically: - GPT-3 (2020): ~4,096 tokens (~3,000 words) - GPT-4 (2023): 8,192–32,768 tokens - Claude 3 (2024): 200,000 tokens (~150,000 words) - Claude 3.5/4 (2024-2025): 200,000+ tokens

Even at 200,000 tokens, a context window is finite. A large enterprise codebase might have millions of lines of code. You cannot fit it all in the context window.

SDD is the discipline of deciding what goes in the context window — and ensuring that what goes in is structured, precise, and sufficient to guide accurate implementation.

💡 Architecture Note: Think of the context window as a whiteboard in a meeting room. An expert developer walking into the meeting without seeing the whiteboard knows nothing about the project. The whiteboard (your specifications, context files, and AGENTS.md) is what you put up before the expert arrives. The quality of the meeting depends entirely on the quality of the whiteboard.

3.5 Hallucinations and How Specifications Prevent Them

Hallucination — when AI generates confident but incorrect information — is one of the most discussed risks of AI systems. In coding contexts, hallucinations manifest as:

  • API hallucinations: Using library functions that don't exist
  • Configuration hallucinations: Generating invalid configuration syntax
  • Business logic hallucinations: Implementing rules that contradict requirements
  • Security hallucinations: "Implementing" security without actually securing anything
  • Integration hallucinations: Generating code that assumes interfaces that don't match existing systems

The root cause of all of these is the same: insufficient or absent specification. When the AI doesn't know the truth, it invents something plausible.

Specifications prevent hallucinations by giving the AI specific, accurate information to work from rather than statistical inference. When you tell the AI exactly which version of a library you're using, exactly what your API returns, and exactly what your business rules require, it has nothing to hallucinate about.

This is not a 100% guarantee — AI models can still produce errors even with complete specifications — but it dramatically reduces the frequency and severity of hallucinations.

3.7 Grounding: The Technical Foundation of Specification Effectiveness

In AI systems, grounding refers to the process of connecting abstract model knowledge to specific, real-world context. A grounded response is one that is anchored to provided facts rather than inferred from training data patterns.

Specifications ground AI responses. When you provide:

## Database Schema
- users table: id (uuid), email (varchar 255, unique), 
  password_hash (varchar 255), created_at (timestamp), 
  updated_at (timestamp)
- sessions table: id (uuid), user_id (uuid FK users.id), 
  token_hash (varchar 255), expires_at (timestamp), 
  revoked (boolean default false)

...the AI is grounded. It knows exactly what the database looks like. It cannot hallucinate column names or data types that don't exist, because you have given it the truth.

Specifications are a grounding mechanism. The more complete and accurate your specifications, the more grounded — and therefore more reliable — your AI's output will be.

3.8 Retrieval-Augmented Specification

In the most sophisticated SDD implementations, specifications are not just loaded wholesale into context windows — they are retrieved on demand based on what the AI currently needs.

This is Retrieval-Augmented Generation (RAG) applied to software specifications. The principle is:

  1. All specifications are indexed and stored in a searchable form
  2. When the AI needs to perform a task, relevant specifications are retrieved automatically
  3. Only the retrieved specifications are loaded into the context window, conserving context space
  4. The AI generates code grounded in precisely the right specifications

Tools like Claude Code (with CLAUDE.md), Cursor (with .cursorrules), and Codeium implement primitive versions of this. The AI loads the most relevant context files based on what it's working on.

More advanced implementations use vector databases to embed specifications and retrieve them semantically — finding relevant context not just by filename but by conceptual similarity to the task at hand.

We will explore this in depth in Chapter 15 (Agentic Systems) and Chapter 19 (MCP).

3.9 The Organizational Case for SDD

Beyond the technical benefits, SDD addresses organizational challenges that become acute in the AI era.

Knowledge concentration: In most organizations, critical system knowledge lives in the heads of a few senior developers. When they leave, knowledge leaves with them. SDD externalizes knowledge into specifications, making it durable and transferable.

AI amplification of knowledge gaps: When a junior developer with partial system knowledge uses an AI assistant, the AI amplifies whatever context is provided — including incomplete or incorrect understanding. SDD ensures that all developers, regardless of experience, are working from the same accurate specification.

Audit and compliance: In regulated industries, specifications provide the audit trail required to demonstrate that software was built to stated requirements. AI-generated code without specifications is extremely difficult to audit.

Onboarding acceleration: New team members can be productive much faster when complete specifications exist. Instead of spending weeks extracting knowledge from senior engineers, they can read specifications and immediately start contributing.

AI model transitions: When you switch from one AI model to another (as the ecosystem evolves rapidly), your specifications come with you. Your investment in SDD is model-agnostic.

4.1 What Is Context Engineering?

Context Engineering is the discipline of designing, structuring, and managing the information provided to AI systems to maximize the quality, accuracy, and reliability of their outputs.

If Prompt Engineering is the art of asking AI the right question, Context Engineering is the science of ensuring AI has everything it needs to answer correctly.

The distinction matters because most AI failures in production software development are not prompt failures — they are context failures. The developer asked the right question; the AI simply didn't have enough information to answer it correctly.

💡 AI Insight: A senior engineer given a clear task but no codebase access, no business context, no architectural guidelines, and no coding standards will still produce mediocre output. The same applies to AI. Context is not optional — it is the prerequisite for quality.

4.2 The Context Engineering Stack

4.3 System Context: Who Is the AI Right Now?

The system context tells the AI what kind of entity it is for this task. In AI assistant terminology, this is typically the "system prompt."

For development tasks, a well-crafted system context establishes: - The AI's role and expertise level - The project's technical stack and conventions - Non-negotiable constraints and policies - The expected output format and style - What the AI should do when uncertain

Weak System Context:

Strong System Context:

You are a senior TypeScript engineer working on TaskFlow, 
a B2B SaaS project management application.

Technical stack:
- Runtime: Node.js 20 LTS
- Language: TypeScript 5.3 (strict mode, no any)
- Framework: NestJS 10 with Express adapter
- ORM: Prisma 5 with PostgreSQL 15
- Authentication: JWT with refresh tokens (stored in httpOnly cookies)
- Testing: Jest + Supertest for integration tests
- Validation: class-validator + class-transformer DTOs

Coding conventions:
- All public functions must have JSDoc comments
- Error handling uses custom AppException class (see src/common/exceptions)
- Database queries MUST go through Repository classes, never direct Prisma calls
- Use Result<T, E> pattern for operations that can fail (see src/common/result.ts)

Security rules (NON-NEGOTIABLE):
- Never store tokens in localStorage
- Always validate user ownership before database mutations
- Rate limit all authentication endpoints

The difference is significant. The second version gives the AI a precise mental model of the project that prevents dozens of small wrong decisions.

4.4 Knowledge Context: What Does the AI Know About Your Domain?

Knowledge context is the information about your business domain, data models, and business rules that the AI needs to generate correct implementations.

Without knowledge context, AI will implement generic versions of features. With it, AI implements your specific version.

Example: Without Knowledge Context

Prompt: "Write a function to calculate the price of an order."

AI generates: price = quantity * unitPrice

This might be correct for a simple store. For your SaaS product with tiered pricing, usage-based components, promotional codes, regional tax rules, and enterprise contract overrides, it is wildly wrong.

Example: With Knowledge Context

## Pricing Engine Domain Knowledge

### Pricing Model
TaskFlow uses a hybrid subscription + usage pricing model:

**Base subscription tiers:**
- Starter: $9/user/month, up to 5 users, 10 projects
- Professional: $25/user/month, unlimited users, unlimited projects
- Enterprise: Custom pricing with annual contract

**Usage-based components (Professional and Enterprise only):**
- Storage: $0.10/GB over included quota (5GB/user)
- API calls: $0.001 per call over 100,000/month
- AI assistant: $0.05 per AI task over 50 included/user/month

- Type PERCENT: Apply percentage discount to subscription only

  - If not VAT registered: Country rate (15-27%)

- Enterprise accounts may have custom per-user pricing
- Enterprise contracts may include committed minimums

4.8 Context Window Management

Context windows are finite. As projects grow larger and specifications more complete, managing what goes into the context window becomes a critical engineering discipline.

Strategies for effective context management:

Strategy 1: Hierarchical loading Load global context (AGENTS.md, architecture) in every session. Load module-specific context only when working in that module. Load feature-specific context only for the specific task.

Strategy 2: Recency bias management AI models give more weight to recent context. Put the most specific, most relevant context closest to the actual task prompt.

Strategy 3: Compression and summarization Long documents can be summarized. A 2000-word architecture document might compress to a 200-word summary without losing the key points needed for most tasks.

Strategy 4: Reference-based context Instead of loading entire files, reference specific functions: "See src/modules/auth/auth.service.ts lines 45-80 for the JWT implementation pattern to follow."

Strategy 5: Explicit context declarations Tell the AI what context you are providing: "For this task, the relevant files are X, Y, and Z. Ignore other parts of the codebase."

Chapter 5: Prompt Engineering — Speaking the Language of AI

5.1 Prompt Engineering vs. Context Engineering

In Chapter 4, we established that Context Engineering is the discipline of what information AI has. Prompt Engineering is the discipline of how you ask for what you need.

Both matter. A perfect context with a poorly crafted prompt produces mediocre output. A perfectly crafted prompt with insufficient context produces confidently wrong output.

The two disciplines are complementary. SDD practitioners need both.

5.2 The Anatomy of an Effective Development Prompt

A well-crafted development prompt has six components:

[ROLE] + [CONTEXT REFERENCE] + [TASK] + [CONSTRAINTS] + [OUTPUT FORMAT] + [VERIFICATION]

Example:

[ROLE] Acting as a senior NestJS engineer familiar with this codebase,

[CONTEXT REF] referencing the project archive specification in FEATURE_ARCHIVE.md 
and following the repository pattern in src/modules/projects/projects.repository.ts,

[TASK] implement the project archive feature including:
- ProjectsRepository methods: archive(id, userId), restore(id, userId)  
- ProjectsService methods: archiveProject, restoreProject with authorization checks
- ProjectsController endpoints: PATCH /projects/:id/archive and /projects/:id/restore
- Prisma migration to add archivedAt and archivedById to Project model

[CONSTRAINTS] Do not modify any existing methods. 
All new methods must include JSDoc. 
Throw ForbiddenException for authorization failures, 
NotFoundException for missing projects. 
Emit project.archived and project.restored events via this.eventEmitter.

[OUTPUT FORMAT] Provide the implementation in this order:
1. Prisma migration SQL
2. Repository methods
3. Service methods  
4. Controller endpoints
5. Unit tests for the service methods

[VERIFICATION] After providing the implementation, list any assumptions you made 
and any edge cases you didn't implement.

This prompt is longer than most developers write. It also produces dramatically better output because every component serves a purpose.

5.4 Prompt Anti-Patterns to Avoid

These prompting approaches consistently produce poor output:

Anti-Pattern 1: The Vague Request

❌ "Add user authentication to the app"
✅ "Implement JWT-based authentication for the REST API following the 
   specification in AUTH_SPEC.md, using the patterns in our existing 
   auth module at src/modules/auth/"

Anti-Pattern 2: The Omnibus Request

❌ "Build the entire user management system including registration, 
   login, profile editing, password reset, email verification, 
   two-factor authentication, social login, session management, 
   and admin user controls"

✅ Break this into individual feature specifications and implement 
   one at a time, in logical dependency order

Anti-Pattern 3: The Assumption Invitation

❌ "Write a function to send emails"

✅ "Write an email service function using the AWS SES SDK v3 
   (already configured in src/config/aws.ts) that sends 
   transactional emails. The function signature must be:
   sendEmail(to: string, template: EmailTemplate, data: Record<string, string>): Promise<Result<void, EmailError>>
   Templates are defined in src/emails/templates/"

Anti-Pattern 4: The Contradictory Request

❌ "Keep it simple but also handle all edge cases and be very 
   comprehensive and add thorough error handling and also 
   make it production-ready"

✅ Define clear, non-contradictory goals. "Simple" and "comprehensive" 
   can coexist if you specify what each means:
   - "Simple API: single function, clear parameters"
   - "Comprehensive: handles timeout, network error, invalid response, rate limiting"

Anti-Pattern 5: The No-Format Request

❌ Just ask for code and accept whatever format the AI chooses

✅ Specify exactly what you need: "Provide the implementation as:
   1. TypeScript file with type definitions
   2. Unit test file
   3. A brief explanation of the main design decisions"

5.5 Chain-of-Thought Prompting for Complex Tasks

For complex architectural tasks, Chain-of-Thought (CoT) prompting dramatically improves output quality by asking the AI to reason through the problem before implementing.

Before implementing the pricing engine, reason through:

1. **Data flow**: Where does pricing data originate? How does it flow 
   through the system? What are the inputs and outputs?

2. **Edge cases**: List every edge case in the pricing specification 
   that could affect the calculation.

3. **Dependencies**: What existing services, repositories, or utilities 
   will this depend on?

4. **Design decisions**: What are the tradeoffs between calculating 
   prices on-the-fly vs. caching pre-calculated prices? 
   Which approach fits our use case better and why?

5. **Testing strategy**: Before writing code, outline how you would 
   test this to ensure pricing is always correct.

After this reasoning, implement the pricing engine.

This approach is especially valuable for: - Architectural decisions - Complex algorithms - Multi-component features - Security-sensitive code - Performance-critical code

5.6 Iterative Prompt Refinement

Effective prompting is rarely single-shot. It is iterative. Here is a refinement loop:

ITERATION 1: Establish basic implementation
Prompt: [Initial specification-driven prompt]
Review: Does it match the spec? What is missing?

ITERATION 2: Address gaps
Prompt: "The implementation is missing X. Add Y following Z pattern."
Review: Is it consistent with the rest of the code?

ITERATION 3: Validate edge cases
Prompt: "What happens when [EDGE CASE]? Update the implementation to handle it."
Review: Is the handling correct per specification?

ITERATION 4: Improve quality
Prompt: "Review this implementation for security issues / performance / conventions."
Review: Are the suggested improvements correct?

ITERATION 5: Generate verification
Prompt: "Generate tests that verify all acceptance criteria in the specification."
Review: Do the tests actually verify the right things?

Each iteration builds on the previous one, progressively converging on a high-quality implementation.

5.7 Prompting for Specification Generation

One of the most powerful uses of AI in SDD is using AI to help write specifications. This seems circular — using AI to write specs to use with AI — but it is genuinely valuable when used correctly.

The key is: you provide the rough requirements, the AI produces a structured specification draft, and you review and correct it.

Here are the rough requirements for a notification system:

- Users should get notified when someone comments on their task
- Users should get notified when they're mentioned with @username
- Users should get notified when a task assigned to them is updated
- Notifications should be real-time
- Users should be able to turn off specific notification types
- There should be an unread count badge

Based on these rough requirements and our tech stack (NestJS, PostgreSQL, 
Socket.IO for real-time), write a complete technical specification for 
the notification system in this format:

1. Business Context (why this exists, who uses it)
2. Data Model (notification schema, preferences schema)
3. Event Architecture (what triggers notifications, event names and payloads)
4. API Specification (all endpoints with full request/response schemas)
5. Real-time Protocol (WebSocket events and connection management)
6. Acceptance Criteria (checkboxes for testing)
7. Edge Cases and Error Scenarios

Flag any ambiguities in the original requirements that need product 
decisions before implementation can begin.

The AI draft will typically be 80-90% correct. Your job is to review it for business accuracy, add constraints the AI didn't know about, and fill in the flagged ambiguities.

This produces specifications faster than writing from scratch, while keeping humans responsible for correctness.

Chapter 6: Spec Engineering — Writing Specifications That Work

6.1 The Art and Science of Specification

Writing a good specification is a skill. Like writing good code, it improves with practice and feedback. And like code, specifications can be good or bad, clear or ambiguous, complete or dangerously incomplete.

Spec Engineering is the discipline of writing specifications that are: - Correct: They accurately describe the desired behavior - Complete: They cover all relevant cases, not just the happy path - Unambiguous: They have only one valid interpretation - Consistent: They don't contradict each other or themselves - Verifiable: The implementation can be tested against them - Machine-readable: AI agents can process them reliably

6.2 The Specification Hierarchy

SDD uses a hierarchy of specifications, from high-level business requirements down to precise implementation contracts:

Level 1: Business Specification
  └── What the business needs and why

Level 2: Product Specification  
  └── What the product does for users

Level 3: Feature Specification
  └── Precise behavioral requirements for specific features

Level 4: Technical Specification
  └── How features are implemented technically

Level 5: API Specification
  └── Exact contracts for service interfaces

Level 6: Data Specification
  └── Schema, constraints, and data models

Each level adds detail and removes ambiguity. You don't need all six levels for every task — a small feature might only need levels 3-5 — but understanding the hierarchy helps you know what level of detail is appropriate.

## 6.5 Specification Anti-Patterns

These patterns consistently produce implementation problems:

### Anti-Pattern 1: The Implicit Happy Path

**Bad:**
> Users can reset their password by entering their email.

**Why it fails**: What happens if the email doesn't exist? What happens after reset? How long is the link valid?

**Good:** Include explicit error paths for every scenario.

### Anti-Pattern 2: The Ambiguous Constraint

**Bad:**
> The system should be fast.

**Why it fails**: "Fast" means different things to different people and cannot be tested.

**Good:**
> - API response time: p95 < 200ms under 100 concurrent users
> - Page load time: Largest Contentful Paint < 2.5s on 4G connection

### Anti-Pattern 3: The Missing Actor

**Bad:**
> Tasks can be deleted.

**Why it fails**: Who can delete? The assignee? The owner? Anyone? Only admins?

**Good:**
> - Task assignees can delete tasks they created within their own projects
> - Project owners can delete any task in their project
> - Workspace admins can delete any task in the workspace
> - Deleted tasks are soft-deleted (archived, not permanently removed)

### Anti-Pattern 4: The Technology Decision Embedded in Behavior

**Bad:**
> The system uses Redis to cache user sessions for 24 hours.

**Why it fails**: This mixes implementation detail (Redis) with behavioral requirement (24-hour sessions). The spec should define the behavior; the technical specification should define the implementation.

**Good:**
> **Behavioral requirement:** User sessions remain valid for 24 hours from last activity.
> **Technical specification:** Implement using Redis sorted sets with TTL refresh on activity.

### Anti-Pattern 5: The Untestable Requirement

**Bad:**
> The UI should be intuitive and user-friendly.

**Why it fails**: Cannot be tested. Cannot be verified. Cannot be implemented with any precision.

**Good:**
> - New users can complete [core task] in under 2 minutes without documentation
> - Error messages explain what went wrong and what to do next
> - All interactive elements have accessible names for screen readers

## 6.6 The CRISP Specification Framework

A useful mnemonic for specification quality:

- **C**omplete: All relevant scenarios covered
- **R**eproducible: Any developer can implement it the same way
- **I**ndependent: Each specification is self-contained
- **S**mall: Specifications cover one coherent feature
- **P**recise: Unambiguous, specific, testable

Run every specification through CRISP before handing it to an AI agent or a developer.

# Chapter 7: Memory Engineering — Teaching AI What to Remember

## 7.1 The Stateless AI Problem

AI coding assistants are stateless. Every session starts with an empty memory. The AI that helped you design your database schema yesterday remembers nothing about it today. The AI that learned your coding conventions last week has forgotten them.

This is one of the most significant practical challenges of AI-assisted development at scale. As your project grows:
- Architectural decisions accumulate
- Coding conventions are established
- Patterns are standardized
- Technology choices are made
- Constraints are discovered

Without a memory system, AI starts every session as a blank slate. With a memory system, it starts every session as an experienced team member who knows the project.

**Memory Engineering** is the discipline of designing systems that provide AI agents with the right historical and contextual knowledge at the right time.

## 7.2 Types of AI Memory

AI memory in development contexts takes several forms:

### Episodic Memory
Specific past events: "We decided on February 12 to use ULID instead of UUID for primary keys because of sortability requirements."

### Semantic Memory
General domain knowledge: "This is a multi-tenant SaaS application. All database queries must filter by tenantId. Never return data across tenant boundaries."

### Procedural Memory
How to do things: "To add a new API endpoint: 1) Define the DTO in the module's dto/ folder, 2) Add the method to the service, 3) Add the controller method with @ApiOperation decorator, 4) Update the OpenAPI spec."

### Working Memory
Current session context: "We've been refactoring the billing module. We've already updated the Service and Repository. We still need to update the Controller and tests."

## 7.3 Memory Implementation Patterns

### Pattern 1: Persistent Documentation Files

The most common memory pattern. Key project decisions and knowledge are written into files in the repository that are automatically loaded into every AI session.

### Pattern 2: Architecture Decision Records (ADRs)

ADRs are a lightweight pattern for recording important architectural decisions. Each decision gets its own file with:

```markdown
# ADR-003: Use ULID Instead of UUID for Primary Keys

## Status: Accepted

## Date: 2024-01-15

## Context
We needed to choose an ID format for database primary keys. Options considered:
- Sequential integers: Simple, but reveals record count (security risk) and doesn't work for distributed systems
- UUID v4: Widely used, random, but poor index performance due to randomness
- UUID v7: Time-ordered UUID, better performance, but not yet widely supported in ORM tooling
- ULID: Universally Unique Lexicographically Sortable Identifier

## Decision
Use ULID for all primary keys.

## Reasoning
1. **Time-ordered**: ULIDs sort lexicographically in creation order, 
   so database indexes remain efficient
2. **Opaque**: Unlike sequential IDs, doesn't reveal business metrics
3. **Short**: 26 characters (vs 36 for UUID with hyphens)
4. **URL-safe**: No special characters
5. **Library support**: @paralleldrive/cuid2 and ulid package both work well

## Consequences
- All primary key columns are VARCHAR(26), not UUID or INTEGER
- ID generation happens in application code (ulid() function), not in the database
- Sort by ID === sort by creation time for most purposes
- Database queries that need "latest records" can use id DESC without a separate created_at index

## Code Example
```typescript
import { ulid } from 'ulid';

const user = await prisma.user.create({
  data: {
    id: ulid(),
    email: input.email,
    // ...
  }
});

AI Instructions

When generating new entities: - Always use ulid() for id generation - Never use crypto.randomUUID() or sequential IDs - VARCHAR(26) for column type - Don't add @default(uuid()) in Prisma schema — generate in app code


When this ADR is loaded into an AI session, every generated entity will correctly use ULIDs without requiring the developer to remember or re-explain the decision.

### API Errors

7.5 Memory Maintenance

Memory systems degrade without maintenance. Outdated memory is worse than no memory — it actively misleads.

Memory maintenance practices:

  1. Update AGENTS.md when architectural decisions change — not after the fact, during the session where the change is made

  2. Add ADRs for every significant decision — the threshold is: "Would a new developer be confused by this choice without context?"

  3. Date-stamp convention changes — so AI sessions can understand which conventions are current vs. historical

  4. Prune obsolete context — remove information about deprecated systems, old frameworks, or replaced patterns

  5. Sprint-level memory — at the start of each sprint, create a sprint context file that captures focus, in-progress items, and key decisions for that sprint

Chapter 8: Repository Engineering — Structure as Specification

8.1 Structure Communicates

Every experienced developer who opens an unfamiliar repository makes dozens of inferences about the project before reading a single line of code. They look at the directory structure, the file naming patterns, the configuration files, and the documentation layout.

Structure communicates. A well-structured repository tells the story of the system — what it does, how it's organized, what technologies it uses, what conventions it follows.

In AI-assisted development, repository structure is a form of specification. AI agents read your repository structure and infer architectural patterns, coding conventions, and organizational principles from it.

Repository Engineering is the discipline of designing repository structure to maximize both human and AI comprehension and productivity.

8.2 The AI-Native Repository

An AI-native repository is designed with the explicit goal of making AI agents as effective as possible. It has several key characteristics:

1. Discoverable context: Critical context is in predictable, findable locations.

3. Self-documenting structure: The directory structure itself communicates architectural decisions.

4. Convention enforcement: Naming and structure are consistent enough that AI can predict where to find things.

5. Progressive disclosure: The top level gives an overview; deeper levels provide detail.

8.6 Naming Conventions as Specification

Consistent naming conventions reduce cognitive load for both humans and AI. When conventions are predictable, AI can generate correct file names, function names, and variable names without being told.

Document your naming conventions explicitly:

## Naming Conventions

### Files
- NestJS modules: `{feature}.module.ts`
- Services: `{feature}.service.ts`
- Repositories: `{feature}.repository.ts`
- Controllers: `{feature}.controller.ts`
- DTOs: `{action}-{resource}.dto.ts` (e.g., `create-task.dto.ts`)
- Guards: `{purpose}.guard.ts` (e.g., `jwt-auth.guard.ts`)
- Specs: `{feature}/SPEC.md`

### Functions and Methods
- Service methods: verb + noun (`createTask`, `findTaskById`, `archiveTask`)
- Repository methods: verb + noun, data-focused (`save`, `findById`, `delete`)
- Controller methods: action-focused (`create`, `findAll`, `findOne`, `update`, `remove`)
- Event handlers: `on` + EventName (`onTaskCreated`, `onProjectArchived`)

### Variables
- Entities: singular noun (`task`, `user`, `project`)
- Collections: plural noun (`tasks`, `users`, `projects`)
- Booleans: `is`, `has`, `can`, `should` prefix (`isActive`, `hasPermission`)
- IDs: entity + `Id` suffix (`taskId`, `userId`, `projectId`)

### Constants
- Upper snake case (`MAX_COMMENT_LENGTH`, `DEFAULT_PAGE_SIZE`)
- In `src/common/constants.ts` for global constants
- In module's `{feature}.constants.ts` for module-specific constants

### Events
- Format: `{domain}.{action}` in present/past tense
- Examples: `task.created`, `project.archived`, `payment.failed`
- Defined in `src/common/events.ts`

When an AI knows these conventions, it generates consistent code without needing to be told each time. This is naming convention as specification.

8.7 The Specification Synchronization Problem

We mentioned in Chapter 2 that specification drift — the divergence between specifications and implementations — is one of the most damaging patterns in software development. Repository engineering provides several mechanisms to combat it.

Mechanism 1: Specs in the Same Repository

When specifications live in the same git repository as code, they are versioned together. A specification change and its corresponding code change go in the same commit. This makes drift visible in code review.

Mechanism 2: BDD Tests as Specifications

When feature specifications are written in Gherkin (BDD) format and connected to test runners, the specification is automatically verified against the implementation. If they diverge, the tests fail.

Mechanism 3: OpenAPI Contract Tests

When your API has an OpenAPI specification, you can run "contract tests" that verify the actual API behavior matches the spec:

# Using Dredd or similar tools
dredd openapi.yaml http://localhost:3000
# Fails if any endpoint's actual behavior differs from the OpenAPI spec

Mechanism 4: Type-Driven Development

In TypeScript projects, types derived from specifications can be used across the codebase. If the specification changes and you update the types, TypeScript's compiler will flag everywhere the implementation doesn't match.

// types/api.ts — Generated from OpenAPI specification
export interface CreateCommentRequest {
  content: string; // min 1, max 2000 characters
}

export interface Comment {
  id: string;
  content: string;

9.1 What Is AGENTS.md?

AGENTS.md is the most important file in any AI-native software project. It is the primary interface between your project and every AI coding assistant that will ever work on it.

Think of it as the comprehensive briefing document you would give to an exceptionally talented contractor on their first day. It tells them: - What this project is and what it does - Who uses it and why - What technology decisions have been made and why - How code is organized and structured - What conventions they must follow - What they must never do - Where to find specific information they need

The name AGENTS.md reflects its purpose in the AI era: it is the configuration document for AI agents working in your repository. Different tools use different names for this file:

Codeium .codeiumrc

We use AGENTS.md as the canonical name throughout this guide because it is tool-agnostic. In practice, you may need to create tool-specific files that reference or extend it.

9.2 The History of AGENTS.md

The concept of a repository-level AI instruction file emerged from the practical experience of developers using early AI coding assistants in 2022-2023.

The pattern was discovered, not designed. Developers noticed that AI assistants performed dramatically better when given project context — and that loading context files manually before every task was tedious. The natural solution was a convention: put project context in a predictable file at the repository root, and AI tools would automatically load it.

Early 2023: Cursor introduced .cursorrules — a markdown file at the repository root that would be automatically injected into every AI interaction. The community immediately embraced it.

Mid 2023: GitHub Copilot added support for copilot-instructions.md in the .github/ directory.

Late 2023: Anthropic introduced CLAUDE.md support in Claude Code, with automatic loading and sophisticated handling of hierarchical context files.

2024: The pattern standardized. Every major AI coding assistant now supports some form of project-level context file. The AGENTS.md convention emerged as a neutral name that can be referenced from tool-specific files.

## 9.9 Section 6: Security Policies

Security policies must be in `AGENTS.md` because AI agents must know them to generate secure code. Security mistakes made by AI are just as dangerous as those made by humans.

```markdown
## Security Policies

These are NON-NEGOTIABLE. Never generate code that violates these policies.

### Authentication & Sessions
- JWT access tokens: 15-minute expiry, stored in memory (not localStorage)
- Refresh tokens: 7-day expiry, stored in httpOnly Secure cookies
- Never store tokens in localStorage or sessionStorage
- Always use the `@Auth()` decorator on protected endpoints
- Session invalidation: refresh token rotation (old token invalidated on use)

### Authorization (CRITICAL)
Every database query that returns user data MUST include ownership validation:
```typescript
// ❌ DANGEROUS — fetches any task

  • Never trust req.body directly — always use typed DTOs

@Post('auth/login')

  • Never log user passwords, tokens, or payment card numbers
  • Never return password hashes in API responses
  • Mask email addresses in logs: user@example.com → u***@example.com
  • Stripe handles all payment data — never touch raw card numbers

Multi-Tenancy Isolation

  • Every resource belongs to a Workspace
  • Every query that returns resources MUST filter by workspaceId (through project membership)

## 9.13 AGENTS.md Anti-Patterns

### Anti-Pattern 1: Too Long, Never Read

An AGENTS.md that is 5,000 words long and covers every conceivable scenario will not be read — by humans or efficiently by AI. Keep it to 1,000–2,000 words at the top level. Use links to deeper documentation for details.

### Anti-Pattern 2: Out of Date

An AGENTS.md that describes a technology stack that changed 3 months ago is worse than none. Outdated context actively misleads. Assign clear ownership for keeping it current.

### Anti-Pattern 3: No Examples

Rules without examples are ambiguous. "Use the repository pattern" without showing what the repository pattern looks like in this codebase leaves interpretation to the AI. Show examples.

### Anti-Pattern 4: Missing Security Policies

If security policies aren't in AGENTS.md, AI agents will use generic security practices — which may not match your requirements. Explicit security policies in AGENTS.md are non-negotiable.

### Anti-Pattern 5: No AI-Specific Instructions

Many AGENTS.md files are just README.md with more detail. The most valuable sections are explicitly for AI: what to do when uncertain, what files not to touch, how to handle ambiguity, expected output formats.

## 9.14 Keeping AGENTS.md Synchronized

AGENTS.md is only valuable when it is accurate. A maintenance process:

1. **Add AGENTS.md review to sprint ceremonies**: At sprint retrospectives, review one section of AGENTS.md and update anything that has changed.

2. **Update in the same PR as the thing it describes**: If you add a new error code pattern, update the "Error Handling" section of AGENTS.md in the same pull request.

3. **Peer review AGENTS.md changes**: Changes to AGENTS.md should be reviewed like code changes — incorrect AGENTS.md can cause systematic AI mistakes.

4. **Version the important parts**: For critical policies (security, data access), include a "Last updated" date so developers and AI know how recent the guidance is.

# Chapter 10: Knowledge Files — Encoding What Your AI Needs to Know

Every developer who joins a project goes through an onboarding process. They read the README. They look at existing code. They ask questions in Slack. They make mistakes, get corrected, and gradually absorb the implicit knowledge that makes the codebase work. This process takes weeks. Sometimes months.

An AI coding assistant gets none of that time.

Every session starts fresh. Every context window is a blank slate. The AI has no memory of yesterday's decisions, last week's refactoring, or the architectural debate that shaped the entire backend. Without deliberate intervention, the AI is perpetually a first-day contractor — technically capable but contextually blind.

Knowledge files solve this problem.

## What Are Knowledge Files?

Knowledge files are structured documents that encode project-specific knowledge in a format that AI assistants can consume efficiently at the start of each session. They are not documentation for humans (though they serve that purpose too). They are **context packages** — curated, dense, and actionable.

The distinction matters. Traditional documentation is written to explain. Knowledge files are written to *inform behavior*. The difference shows up in how they're structured:

**Traditional README (written for humans):**
```markdown
## Getting Started

This project is a task management SaaS. To get started, install Node.js and 
run `npm install`. See the contributing guide for more details.

Knowledge file (written for AI):

## Project: TaskFlow

TYPE: B2B SaaS, multi-tenant task management
STACK: NestJS 10, PostgreSQL 16, Prisma ORM, Redis 7, React 18, TypeScript 5
ARCHITECTURE: Modular monolith, feature-based folder structure
TENANT ISOLATION: Row-level security via tenant_id on all tables
AUTH: JWT + refresh tokens; roles: OWNER, ADMIN, MEMBER, VIEWER
API STYLE: RESTful with consistent envelope: { data, meta, error }
CRITICAL CONSTRAINTS:
  - Never use `any` type
  - All DB queries must include tenant_id filter
  - Services must not import from other feature modules directly — use events
  - Dates always stored as UTC, displayed in user timezone

The second version takes three seconds to read but gives an AI everything it needs to generate correctly-scoped, architecturally-consistent code.

Chapter 11: Planning Files — Turning Intent into Executable Work

Software development fails in two ways: building the wrong thing, and building the right thing poorly. Knowledge files address the second failure mode. Planning files address the first.

A planning file is a structured document that captures what needs to be built, why it matters, and how to verify it's done correctly — before any code is written. In the context of AI-assisted development, planning files are the difference between an AI that generates a plausible implementation and one that generates the right implementation.

The Planning Hierarchy

Planning exists at multiple timescales. Effective SDD teams maintain a hierarchy of planning artifacts:

Strategic     │  ROADMAP.md         → 6-12 month direction
              │  PRODUCT.md         → Product vision and priorities
              │
Tactical      │  SPRINT.md          → 2-week sprint plan
              │  MILESTONE.md       → Feature milestone breakdown
              │
Operational   │  FEATURE.md         → Individual feature specification
              │  TASK.md            → Implementation task detail
              │
Session       │  SESSION.md         → Today's coding session goals
              │  PROMPT.md          → Specific AI prompt context

Each level feeds into the next. The ROADMAP informs the SPRINT. The SPRINT informs the FEATURE. The FEATURE informs the SESSION. This cascade ensures that each AI coding session is anchored to real business intent.

Chapter 12: Clean Architecture in AI-Native Projects

Architecture is not about code organization. Architecture is about decisions — which things can change independently, which things must change together, and which things must never change at all.

Clean Architecture, introduced by Robert C. Martin, answers these questions by organizing code into layers where dependencies point inward. Business logic sits at the center, untouched by frameworks, databases, or external services. Infrastructure sits on the outside, easily replaceable.

In an AI-native project, Clean Architecture becomes even more valuable. When an AI assistant generates code, it tends to reach for the nearest dependency — importing database clients into business logic, mixing HTTP concerns with domain rules, coupling services to specific frameworks. Well-defined architecture specifications constrain this behavior, producing code that's genuinely maintainable.

The Clean Architecture Model

┌─────────────────────────────────────────────┐
│                   Frameworks                 │
│    (Express, NestJS, React, PostgreSQL)     │
│  ┌──────────────────────────────────────┐   │
│  │          Interface Adapters          │   │
│  │  (Controllers, Presenters, Gateways) │   │
│  │  ┌────────────────────────────────┐  │   │
│  │  │       Application Layer        │  │   │
│  │  │  (Use Cases, Application Logic)│  │   │
│  │  │  ┌──────────────────────────┐  │  │   │
│  │  │  │      Domain Layer        │  │  │   │
│  │  │  │  (Entities, Value Objects│  │  │   │
│  │  │  │   Domain Services, Rules)│  │  │   │
│  │  │  └──────────────────────────┘  │  │   │
│  │  └────────────────────────────────┘  │   │
│  └──────────────────────────────────────┘   │
└─────────────────────────────────────────────┘

The Dependency Rule: Source code dependencies must point inward. The domain layer knows nothing about the application layer. The application layer knows nothing about the interface adapters. The interface adapters know nothing about the framework layer.

This rule has a critical consequence: your business logic doesn't know whether it's running in a web server, a CLI tool, a background job, or a test.

The Domain Layer

Domain entities encode business rules directly in code. They're plain TypeScript classes with no framework dependencies:

// domain/entities/task.entity.ts

export class Task {
  readonly id: string;
  private _status: TaskStatus;
  private _title: string;
  private _completedAt: Date | null;

  constructor(props: TaskProps) {
    this.id = props.id;
    this._status = props.status;
    this._title = props.title;
    this._completedAt = props.completedAt ?? null;

    this.validate();
  }

  // Business method: encodes the rule that only IN_REVIEW tasks can be completed
  complete(): void {
    if (this._status !== TaskStatus.IN_REVIEW) {
      throw new DomainError(
        `Task cannot be completed from status: ${this._status}. ` +
        `Task must be IN_REVIEW first.`
      );
    }

    this._status = TaskStatus.DONE;
    this._completedAt = new Date();
  }

  // Business method: validates assignment rules
  assignTo(userId: string, assignerRole: WorkspaceRole): void {
    if (assignerRole === WorkspaceRole.VIEWER) {
      throw new DomainError('Viewers cannot assign tasks');
    }

    this._assigneeId = userId;
  }

  private validate(): void {
    if (!this._title || this._title.trim().length === 0) {
      throw new DomainError('Task title cannot be empty');
    }
    if (this._title.length > 500) {
      throw new DomainError('Task title cannot exceed 500 characters');
    }
  }

  // Getters (no setters — use methods with business meaning)
  get status(): TaskStatus { return this._status; }
  get title(): string { return this._title; }
  get completedAt(): Date | null { return this._completedAt; }
}

// application/use-cases/complete-task.use-case.ts

export class CompleteTaskUseCase {
  constructor(
    private readonly taskRepo: TaskRepository,        // Interface, not implementation
    private readonly notificationService: NotificationService, // Interface
    private readonly events: DomainEventBus,
  ) {}

  async execute(command: CompleteTaskCommand): Promise<CompleteTaskResult> {
    // Load aggregate
    const task = await this.taskRepo.findById(
      command.taskId, 
      command.workspaceId
    );

    if (!task) {
      throw new TaskNotFoundError(command.taskId);
    }

    // Execute domain logic (business rules enforced by entity)
    task.complete();

    // Persist
    const savedTask = await this.taskRepo.save(task);

    // Side effects (notifications, events)
    await this.events.publish(new TaskCompletedEvent({
      taskId: task.id,
      workspaceId: command.workspaceId,
      completedBy: command.userId,
      completedAt: task.completedAt!,
    }));

    return { task: savedTask };
  }
}

export class PrismaTaskRepository implements TaskRepository {
  constructor(private readonly prisma: PrismaService) {}

  async findById(id: string, workspaceId: string): Promise<Task | null> {
    const record = await this.prisma.task.findFirst({
      where: { id, workspaceId, deletedAt: null },
    });

    if (!record) return null;

    // Map database record to domain entity
    return this.toDomain(record);
  }

  async save(task: Task): Promise<Task> {
    const data = this.toPersistence(task);

    const record = await this.prisma.task.upsert({
      where: { id: task.id },
      create: data,
      update: data,
    });

    return this.toDomain(record);
  }

  // Mappers: convert between database records and domain entities
  private toDomain(record: PrismaTask): Task {
    return new Task({
      id: record.id,
      title: record.title,
      status: record.status as TaskStatus,
      completedAt: record.completedAt,
      // ... other fields
    });
  }

  private toPersistence(task: Task): Prisma.TaskCreateInput {
    return {
      id: task.id,
      title: task.title,
      status: task.status,
      completedAt: task.completedAt,
      // ... other fields
    };

Chapter 13: Hexagonal Architecture and Feature-Based Structure

Clean Architecture describes a principle. Hexagonal Architecture (also called Ports and Adapters) describes a concrete implementation pattern. Both express the same core idea — isolate business logic from infrastructure — but hexagonal architecture is often more practical for teams building modern APIs.

Ports and Adapters

In Hexagonal Architecture, the application exposes ports (interfaces) through which all communication happens. Adapters implement these ports for specific technologies.

                    ┌────────────────────────────────┐
                    │         Application Core        │
  HTTP Request ───→ │                                │ ───→ PostgreSQL
                    │   Ports:                        │
  GraphQL ────────→ │   - TaskRepository (Port)       │ ───→ Redis
                    │   - NotificationPort (Port)     │
  CLI Command ────→ │   - SearchPort (Port)           │ ───→ Elasticsearch
                    │                                 │
  Message Queue ──→ │   Core Business Logic           │ ───→ Email Service
                    │                                 │
                    └────────────────────────────────┘

         Left side = Driving Adapters (input)
         Right side = Driven Adapters (output)

The left side (driving adapters) drives the application: HTTP controllers, GraphQL resolvers, CLI commands, message queue consumers. The right side (driven adapters) is driven by the application: databases, caches, email providers, external APIs.

This symmetry is powerful. Replacing PostgreSQL with MySQL is a driven adapter swap — the core application is unchanged.

Chapter 14: Event-Driven Architecture and Agentic Systems

Software architectures are increasingly event-driven. Instead of components calling each other directly (tight coupling), they publish events and subscribe to the events they care about. This decoupling is essential for scalability, resilience, and — critically for AI-native projects — the ability to add new behaviors without modifying existing code.

Agentic systems are inherently event-driven. An agent observes events, decides what to do, takes actions, and those actions produce more events. Understanding this parallelism is key to designing AI systems that compose.

Why Events Matter in AI-Native Systems

Consider the task completion flow in TaskFlow:

Without Events (Coupled):

// tasks.service.ts
async completeTask(taskId: string, userId: string): Promise<Task> {
  const task = await this.taskRepo.complete(taskId);

  // Direct coupling to every downstream concern:
  await this.notificationService.sendTaskCompleted(task);
  await this.analyticsService.recordCompletion(task, userId);
  await this.sprintService.updateVelocity(task);
  await this.aiSuggestionService.updateModel(task);  // NEW: had to add here

  return task;
}

Every new capability requires modifying the service. Add AI suggestions, modify the service. Add webhooks, modify the service. Add Slack notifications, modify the service.

With Events (Decoupled):

// tasks.service.ts
async completeTask(taskId: string, userId: string): Promise<Task> {
  const task = await this.taskRepo.complete(taskId);

  // Publish ONE event — no knowledge of downstream consumers
  this.events.emit('task.completed', new TaskCompletedEvent({
    taskId: task.id,
    workspaceId: task.workspaceId,
    completedBy: userId,
    completedAt: new Date(),
  }));

  return task;
}

// Notification subscriber (separate file, separate concern)
@OnEvent('task.completed')
async handleTaskCompleted(event: TaskCompletedEvent): Promise<void> {
  await this.sendNotification(event);
}

// AI suggestion subscriber (added later, no changes to task service)
@OnEvent('task.completed')  
async updateAIModel(event: TaskCompletedEvent): Promise<void> {
  await this.aiSuggestionModel.recordCompletion(event);
}

New capabilities are additions, not modifications. This is the Open-Closed Principle at the architecture level.

Introduction to Agentic Architectures

An AI agent is not a chat interface. An agent is a system that:

  1. Perceives an environment (reads files, calls APIs, receives events)
  2. Reasons about what to do next (LLM processing)
  3. Acts on its decisions (writes code, calls tools, sends messages)
  4. Observes the result and repeats

This loop — Perceive → Reason → Act → Observe — is the foundation of agentic software engineering. It's also, functionally, what a senior developer does when they receive a bug report.

┌──────────────────────────────────────────────────────┐
│                     Agent Loop                       │
│                                                      │
│   Perception          Reasoning           Action     │
│  ┌──────────┐       ┌──────────┐       ┌──────────┐ │
│  │ Read     │       │          │       │ Write    │ │
│  │ Files    │──────→│   LLM    │──────→│ Files    │ │
│  │ Run Tests│       │          │       │ Run Code │ │
│  │ Call APIs│←──────│ Reasoning│←──────│ Call APIs│ │
│  └──────────┘       └──────────┘       └──────────┘ │
│       ↑                                     │        │
│       └─────────────────────────────────────┘        │
│                     Observe results                  │
└──────────────────────────────────────────────────────┘

Agentic systems are the subject of the next major section of this guide (Chapters 15-17). What's important to establish here is how event-driven architecture and agentic patterns reinforce each other:

  • Events give agents natural observation points
  • Agent actions produce events that other agents (or services) react to
  • The event log becomes the agent's memory of what happened
  • Events provide a natural audit trail for agentic actions

Most developers today operate in the Assisted to Augmented range. Agentic development is rapidly becoming practical. Fully autonomous development — where AI operates without human oversight — is emerging for constrained, well-specified tasks.

SDD is the framework that makes agentic development safe and effective. Without precise specifications, agentic systems produce plausible but incorrect implementations. With precise specifications, they produce correct, consistent, maintainable code.

What Makes a Good Agentic Task?

Not all tasks are equal candidates for agentic execution. The right tasks for agents are:

Well-specified: The agent knows exactly what success looks like. Ambiguity causes agents to make assumptions, which cascade into wrong implementations.

Bounded: The agent has a clear start and end state. "Refactor the entire codebase" is not agentic-ready. "Rename all instances of userId to user_id in the tasks module" is.

Verifiable: Success can be checked programmatically. Tests that pass, linting that cleans, types that compile — these give agents feedback loops.

Reversible: Agent actions can be undone if they go wrong. Git branches, staging environments, and database transactions provide the safety net.

## Agentic Task Specification Template

TASK: {one-line description}
SCOPE: {which files/modules are affected}
SUCCESS_CRITERIA: {how to verify completion}
CONSTRAINTS: {what NOT to change}
VERIFICATION: {command to run to verify}
ROLLBACK: {how to undo if needed}

TASK: Add input validation to all task creation DTOs

  - Do not add new dependencies

The ReAct Pattern for AI Agents

The ReAct (Reasoning + Acting) pattern is the dominant framework for implementing AI agents. It interleaves reasoning steps with action steps, giving agents the ability to plan, execute, observe, and adjust.

Test-Driven Agentic Development (TDAD)

One of the most powerful patterns in agentic development is having the agent write tests before implementation — then implement until the tests pass. This inverts the typical AI workflow (generate code, hope it's right) into a verifiable loop (define correctness, generate code that satisfies it).

1. Human provides feature specification
2. Agent reads specification
3. Agent generates test suite (without implementation)
4. Human reviews tests — are they correct? Do they capture the spec?
5. Human approves
6. Agent implements until all tests pass
7. Agent presents diff for final review

This pattern is particularly effective because: - Tests formalize the specification (if you can't write the test, the spec is unclear) - The agent has an objective success criterion (tests passing) - Human review of tests is faster than review of implementation - Failing tests give the agent immediate, specific feedback

16.1 Why Multiple Agents?

A single AI agent, even a capable one, faces practical limitations: - Context window size limits how much it can "see" at once - Single-agent systems have no checks on their own reasoning - Complex tasks may require different "modes" of thinking - Long-running tasks risk quality degradation as context fills

Multi-agent systems address these limitations by distributing work across specialized agents, each focused on a specific aspect of the task.

16.3 A Real Multi-Agent Workflow

Here is a concrete multi-agent workflow for implementing a new feature:

// orchestrator.agent.ts
async function implementFeature(featureSpec: string): Promise<ImplementationResult> {

  // PHASE 1: Architecture Review
  const architectureDecision = await architectAgent.review({
    spec: featureSpec,
    existingArchitecture: await readFile('ARCHITECTURE.md'),
    constraints: await readFile('AGENTS.md'),
    task: 'Review this feature spec and produce an implementation plan with file structure'
  });

  await humanApproval('Architecture Plan', architectureDecision.plan);

  // PHASE 2: Test Generation (TDD approach)
  const tests = await testerAgent.generateTests({
    spec: featureSpec,
    plan: architectureDecision.plan,
    testExamples: await readFile('src/modules/tasks/tasks.service.spec.ts'),
    task: 'Generate failing tests that encode all acceptance criteria'
  });

  await humanApproval('Tests', tests);

  // PHASE 3: Implementation
  const implementation = await coderAgent.implement({
    spec: featureSpec,
    plan: architectureDecision.plan,
    tests: tests,
    agentsContext: await readFile('AGENTS.md'),
    task: 'Implement the feature so all tests pass'
  });

  // PHASE 4: Review
  const review = await reviewerAgent.review({
    implementation: implementation,
    spec: featureSpec,
    securityPolicies: await readFile('SECURITY.md'),
    task: 'Review for correctness, security, and convention compliance'
  });

  if (review.issues.length > 0) {
    // Send back to coder with review feedback
    return implementFeature_withFeedback(featureSpec, implementation, review);
  }

  return { implementation, tests, review };
}

16.4 Specification-Driven Agent Coordination

The key to reliable multi-agent workflows is that all agents work from the same specifications. This prevents contradictions where the architect designs one thing, the coder implements another, and the tester verifies a third.

In an SDD multi-agent system:

  1. Specifications are the shared truth — all agents read the same spec files
  2. Agents communicate through artifacts — not through conversation; one agent produces files that the next agent reads
  3. Checkpoints create verifiable state — each agent phase produces output that can be reviewed before proceeding
  4. Failure triggers re-specification — if review fails, the spec is updated, not just the code

17.4 Self-Healing Agents

The most advanced agent pattern is the Self-Healing Agent — one that can detect when something is broken and autonomously attempt to fix it.

stateDiagram-v2
    [*] --> Monitor
    Monitor --> Alert: Test Failure Detected
    Alert --> Analyze: Analyze Failure
    Analyze --> Plan: Generate Fix Plan
    Plan --> Implement: Implement Fix
    Implement --> Verify: Run Tests
    Verify --> Monitor: All Tests Pass
    Verify --> HumanEscalation: Still Failing After 3 Attempts
    HumanEscalation --> [*]

// self-healing-agent.ts
async function selfHealingLoop(context: HealingContext): Promise<HealingResult> {
  const MAX_ATTEMPTS = 3;

  for (let attempt = 1; attempt <= MAX_ATTEMPTS; attempt++) {
    // Run the test suite
    const testResults = await runTests();

    if (testResults.allPassed) {
      return { success: true, attempts: attempt };
    }

    // Analyze failures
    const analysis = await analyzeAgent.analyze({
      failures: testResults.failures,
      codeContext: await gatherRelevantCode(testResults.failures),
      spec: await readSpec(testResults.failures)
    });

    // Generate and apply fix
    const fix = await coderAgent.fix({
      analysis: analysis,
      agentsContext: await readFile('AGENTS.md'),
    });

    await applyFix(fix);

    // Log the attempt
    console.log(`Healing attempt ${attempt}: Fixed ${fix.filesChanged.length} files`);
  }

  // Escalate to human after MAX_ATTEMPTS
  await notifyHuman({
    message: `Self-healing failed after ${MAX_ATTEMPTS} attempts`,
    failures: await runTests(),
    attempts: collectAttemptHistory()
  });

  return { success: false, attempts: MAX_ATTEMPTS };
}

Self-healing agents work most reliably when: 1. The failing tests are specification-derived (they describe what should happen, not what does happen) 2. The specifications in context are accurate and complete 3. The scope of the fix is bounded (fixing one module, not the whole system) 4. Human escalation is always available as a fallback

Chapter 19: Model Context Protocol (MCP)

19.1 What Is MCP?

The Model Context Protocol (MCP) is an open protocol developed by Anthropic that standardizes how AI models connect to external data sources and tools. It solves one of the fundamental limitations of AI systems: the inability to access real-world data and services beyond their training data.

MCP creates a standardized interface between: - MCP Hosts: AI models and assistants (Claude, Claude Code, Cursor, etc.) - MCP Servers: Tools and data sources (databases, APIs, file systems, etc.)

┌─────────────────┐         ┌─────────────────────┐
│   AI Model      │◄───────►│   MCP Server        │
│  (MCP Client)   │  MCP    │                     │
│                 │ Protocol│  • GitHub API        │
│  Claude Code    │         │  • PostgreSQL        │
│  Cursor         │         │  • File System       │
│  VS Code        │         │  • Jira              │
└─────────────────┘         │  • Slack             │
                            │  • Custom APIs       │
                            └─────────────────────┘

19.2 MCP in SDD Context

For SDD practitioners, MCP enables something powerful: specifications become live connections to the systems they describe.

Instead of an AI reading a static specification document, it can use MCP to: - Query the actual database schema (not a documented version that might be outdated) - Check the actual API behavior (not a specification that might have drifted) - Read current deployment status (not outdated infrastructure docs) - Access real project state (not a summary written last month)

This solves the specification drift problem at its root: if specifications are connected to the systems they describe, they cannot drift.

19.3 Building an MCP Server

An MCP server is a simple Node.js (or Python) process that exposes tools and resources through the MCP protocol.

// custom-mcp-server.ts
import { Server } from '@modelcontextprotocol/sdk/server/index.js';
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js';
import { PrismaClient } from '@prisma/client';

const prisma = new PrismaClient();
const server = new Server(
  { name: 'taskflow-database', version: '1.0.0' },
  { capabilities: { tools: {} } }
);

// Define available tools
server.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: [
    {
      name: 'get_schema',
      description: 'Get the current database schema for a model',
      inputSchema: {
        type: 'object',
        properties: {
          model: { type: 'string', description: 'The Prisma model name' }
        },
        required: ['model']
      }
    },
    {
      name: 'query_data',
      description: 'Run a read-only query against the database',
      inputSchema: {
        type: 'object',
        properties: {
          query: { type: 'string', description: 'SQL SELECT query (read-only)' }
        },
        required: ['query']
      }
    },
    {
      name: 'get_api_spec',
      description: 'Get the OpenAPI specification for an endpoint',
      inputSchema: {
        type: 'object',
        properties: {
          path: { type: 'string', description: 'API path (e.g., /api/tasks)' },
          method: { type: 'string', description: 'HTTP method' }
        },
        required: ['path', 'method']
      }
    }
  ]
}));

// Handle tool calls
server.setRequestHandler(CallToolRequestSchema, async (request) => {
  switch (request.params.name) {
    case 'get_schema':
      const model = await prisma.$queryRaw`
        SELECT column_name, data_type, is_nullable
        FROM information_schema.columns
        WHERE table_name = ${request.params.arguments.model}
      `;
      return { content: [{ type: 'text', text: JSON.stringify(model, null, 2) }] };

    case 'query_data':
      // SECURITY: Only allow SELECT statements
      const query = request.params.arguments.query;
      if (!query.trim().toUpperCase().startsWith('SELECT')) {
        throw new Error('Only SELECT queries are allowed');
      }
      const results = await prisma.$queryRawUnsafe(query);
      return { content: [{ type: 'text', text: JSON.stringify(results, null, 2) }] };
  }
});

// Start the server
const transport = new StdioServerTransport();
await server.connect(transport);

Chapter 20: Testing in an AI-Assisted World

20.1 How AI Changes Testing

Testing was already important before AI. In the AI era, it becomes mission-critical. Here is why:

When a human developer makes a mistake, they typically make one mistake in one place. They then notice it in testing, fix it, and move on.

When an AI agent makes a systematic misunderstanding — misinterpreting a requirement, using an incorrect pattern, or applying a wrong assumption — it applies that mistake consistently across every file it generates. One wrong assumption becomes dozens of files with the same problem.

Good test coverage is the mechanism that catches systematic AI errors before they reach production. Tests derived from specifications — not from implementations — are the gold standard.

20.2 Specification-Derived Tests

The most valuable tests in an SDD project are written from the specification, not by reading the implementation. Here is the principle:

Tests written from the spec verify that the implementation matches the spec. Tests written from the implementation verify that the implementation is self-consistent.

The second type is far less valuable. It will pass even when the implementation is completely wrong — as long as it's consistently wrong.

// GOOD: Test derived from specification
// The spec says: "Rate limit: Maximum 3 reset requests per email per hour"
it('should return 429 after 3 password reset requests in an hour', async () => {
  const email = 'user@example.com';

  // First 3 requests succeed
  for (let i = 0; i < 3; i++) {
    const response = await request(app)
      .post('/api/auth/forgot-password')
      .send({ email });
    expect(response.status).toBe(200);
  }

  // 4th request within the same hour should be rate limited
  const response = await request(app)
    .post('/api/auth/forgot-password')
    .send({ email });
  expect(response.status).toBe(429);
});

// BAD: Test written from reading the implementation
// The test checks the implementation details, not the specification
it('should call rateLimitService.increment() with the email', async () => {
  // This test verifies implementation details, not business requirements
  // If we change the rate limiting implementation, this test breaks
  // even if the behavior is still correct
});

The most rigorous approach to AI-assisted development combines TDD with specification-driven design:

// packages/api/src/services/tasks.service.ts

import { PrismaClient, Task, TaskStatus } from '@prisma/client';
import { AppError } from '../errors/app-error';
import { CreateTaskDto, UpdateTaskDto } from '../types/tasks.types';
import { ulid } from 'ulid';

export class TasksService {
  constructor(private readonly prisma: PrismaClient) {}

  async createTask(
    userId: string,
    workspaceId: string,
    dto: CreateTaskDto
  ): Promise<Task> {
    // Verify user is a workspace member
    const membership = await this.prisma.workspaceMember.findUnique({
      where: { workspaceId_userId: { workspaceId, userId } }
    });

    if (!membership) {
      throw new AppError('FORBIDDEN', 'Not a workspace member', 403);
    }

    // If assignee is provided, verify they're a workspace member too
    if (dto.assigneeId) {
      const assigneeMembership = await this.prisma.workspaceMember.findUnique({
        where: { workspaceId_userId: { workspaceId, userId: dto.assigneeId } }
      });
      if (!assigneeMembership) {
        throw new AppError('INVALID_INPUT', 'Assignee is not a workspace member', 400);
      }
    }

    return this.prisma.task.create({
      data: {
        id: ulid(),
        title: dto.title,
        description: dto.description,
        status: TaskStatus.TODO,
        priority: dto.priority ?? 'MEDIUM',
        dueDate: dto.dueDate ? new Date(dto.dueDate) : null,
        workspaceId,
        assigneeId: dto.assigneeId,
        createdById: userId,
      },
      include: {
        assignee: { select: { id: true, name: true, email: true } },
        tags: true,
      }
    });
  }

  async findAll(
    userId: string,
    workspaceId: string,
    options: { status?: TaskStatus; assigneeId?: string; page?: number; limit?: number }
  ): Promise<{ tasks: Task[]; total: number }> {
    // Verify membership
    await this.assertMembership(userId, workspaceId);

    const page = options.page ?? 1;
    const limit = Math.min(options.limit ?? 20, 100); // Cap at 100
    const skip = (page - 1) * limit;

    const where = {
      workspaceId,
      deletedAt: null, // Exclude soft-deleted
      ...(options.status && { status: options.status }),
      ...(options.assigneeId && { assigneeId: options.assigneeId }),
    };

    const [tasks, total] = await this.prisma.$transaction([
      this.prisma.task.findMany({
        where,
        include: {
          assignee: { select: { id: true, name: true } },
          tags: true,
          _count: { select: { comments: true } }
        },
        orderBy: [
          { priority: 'desc' },
          { dueDate: 'asc' },
          { createdAt: 'desc' }
        ],
        skip,
        take: limit,
      }),
      this.prisma.task.count({ where })
    ]);

    return { tasks, total };
  }

  async updateTask(
    userId: string,
    workspaceId: string,
    taskId: string,
    dto: UpdateTaskDto
  ): Promise<Task> {
    const task = await this.findTaskInWorkspace(taskId, workspaceId);

    // Authorization: assignee or workspace admin/owner can edit
    const membership = await this.prisma.workspaceMember.findUnique({
      where: { workspaceId_userId: { workspaceId, userId } }
    });

    const canEdit = task.assigneeId === userId || 
                    task.createdById === userId ||
                    membership?.role === 'ADMIN' ||
                    membership?.role === 'OWNER';

    if (!canEdit) {
      throw new AppError('FORBIDDEN', 'You cannot edit this task', 403);
    }

    const completedAt = dto.status === 'DONE' && task.status !== 'DONE'
      ? new Date()
      : dto.status !== 'DONE' && task.status === 'DONE'
        ? null
        : undefined; // Don't change completedAt if status didn't change

    return this.prisma.task.update({
      where: { id: taskId },
      data: {
        ...(dto.title !== undefined && { title: dto.title }),
        ...(dto.description !== undefined && { description: dto.description }),
        ...(dto.status !== undefined && { status: dto.status }),
        ...(dto.priority !== undefined && { priority: dto.priority }),
        ...(dto.dueDate !== undefined && { dueDate: dto.dueDate ? new Date(dto.dueDate) : null }),
        ...(dto.assigneeId !== undefined && { assigneeId: dto.assigneeId }),
        ...(completedAt !== undefined && { completedAt }),
      },
      include: {
        assignee: { select: { id: true, name: true } },
        tags: true
      }
    });
  }

  private async findTaskInWorkspace(taskId: string, workspaceId: string): Promise<Task> {
    const task = await this.prisma.task.findFirst({
      where: { id: taskId, workspaceId, deletedAt: null }
    });
    if (!task) {
      throw new AppError('NOT_FOUND', 'Task not found', 404);
    }
    return task;
  }

  private async assertMembership(userId: string, workspaceId: string): Promise<void> {
    const membership = await this.prisma.workspaceMember.findUnique({
      where: { workspaceId_userId: { workspaceId, userId } }
    });
    if (!membership) {
      throw new AppError('FORBIDDEN', 'Not a workspace member', 403);
    }
  }

#### Grounding (Critical)
The AI MUST only answer questions using information from the retrieved documents.
If the answer is not in the retrieved documents, the AI must say so and offer

NEVER hallucinate answers that aren't in the documents.

The AI must adopt the company's configured tone:
- PROFESSIONAL: Formal, no contractions, business language
- FRIENDLY: Casual, warm, use of "you" and "we"  
- TECHNICAL: Precise, use domain terminology, assume technical knowledge

#### Handoff Triggers
The AI MUST offer human handoff when:
- It says it doesn't know the answer twice in one conversation
- The user expresses frustration (detected by sentiment)
- The user explicitly asks to speak with a human
- The question involves personal account data the AI cannot access

### Testing AI Behavior

#### Deterministic Tests
For deterministic behavior, test with fixed seed or temperature=0:
- Citation format: parse AI response for citation patterns
- Handoff trigger: send "I want to speak with a human" → verify handoff offered
- Out-of-scope: send question about unrelated topic → verify "I don't know" response

#### Evaluation Tests (Non-Deterministic)
For quality, use LLM-as-judge approach:
- Send 50 sample questions from real customer support
- Have Claude-judge evaluate: Is the response grounded? Is it accurate? Is it helpful?
- Require >90% pass rate before any model or prompt change goes to production

@Injectable()
export class PromptService {
  constructor(private readonly prisma: PrismaClient) {}

  async getActivePrompt(workspaceId: string, promptType: PromptType): Promise<PromptVersion> {
    const prompt = await this.prisma.promptVersion.findFirst({
      where: { workspaceId, type: promptType, isActive: true },
      orderBy: { createdAt: 'desc' }
    });

    if (!prompt) {
      // Fall back to global default
      return this.getDefaultPrompt(promptType);
    }

    return prompt;
  }

  buildSystemPrompt(template: string, variables: Record<string, string>): string {
    // Replace {{variable}} placeholders
    return template.replace(/\{\{(\w+)\}\}/g, (_, key) => variables[key] ?? '');
  }

  async buildRAGPrompt(
    workspaceId: string,
    question: string,
    retrievedChunks: DocumentChunk[],
    conversationHistory: Message[],
  ): Promise<AnthropicMessage[]> {
    const promptTemplate = await this.getActivePrompt(workspaceId, PromptType.SUPPORT);
    const workspace = await this.prisma.workspace.findUnique({ where: { id: workspaceId } });

    const systemPrompt = this.buildSystemPrompt(promptTemplate.systemTemplate, {
      companyName: workspace.name,
      tone: workspace.aiTone,
      currentDate: new Date().toISOString().split('T')[0],
    });

    const contextBlock = retrievedChunks
      .map((chunk, i) => `[Source ${i+1}: ${chunk.metadata.title}]\n${chunk.content}`)
      .join('\n\n---\n\n');

    // Build conversation history (last 10 exchanges)
    const history: AnthropicMessage[] = conversationHistory
      .slice(-20) // Last 20 messages = 10 exchanges
      .map(msg => ({
        role: msg.role === 'USER' ? 'user' : 'assistant',
        content: msg.content
      }));

    return [
      // System message sets behavior
      // Context injected as a user message before the actual question
      ...history.slice(0, -1), // Previous history
      {
        role: 'user',
        content: `<context>\n${contextBlock}\n</context>\n\n${question}`
      }
    ];
  }

Chapter 26: Common Failures and How to Avoid Them

26.1 Failure Catalog

Failure 1: The Drifting Specification

What happens: The specification is written but never updated after implementation. Over time, the specification describes a system that no longer exists.

Symptoms: - AI generates code based on spec that conflicts with existing implementation - New developers are confused by discrepancies - Tests based on the spec fail against the implementation

Prevention: - Update spec in the same PR as implementation changes - Add spec review to definition of done - Use BDD tests as executable specs (can't drift)

Failure 2: The Over-Specified System

What happens: Every implementation detail is specified, leaving no room for engineering judgment. Specifications become rigid and brittle.

Symptoms: - Specifications describe how to implement, not what to achieve - Specifications break when libraries update - Developers and AI spend more time matching specifications than solving problems

Prevention: - Specify behavior and constraints, not implementation - Separate "what" (specification) from "how" (technical decision) - Allow AI to choose implementation approach within specification constraints

Failure 3: The Context Flood

What happens: Too much context is loaded into AI sessions. The AI is overwhelmed and performs worse than with less context.

Symptoms: - AI responses become generic when given very large context files - AI ignores specific instructions buried in long documents - Session quality degrades for complex projects

Prevention: - Keep AGENTS.md focused (under 2,000 words) - Use hierarchical context files (load only what's needed) - Put most critical information first in context files - Use MCP for dynamic, on-demand context retrieval

Failure 4: The Security Theater Specification

What happens: Security requirements are specified vaguely, and AI implements superficial security that appears correct but is easily bypassed.

Examples: - "Implement authentication" → AI adds JWT but doesn't validate expiry - "Protect user data" → AI adds auth middleware but not ownership checks - "Prevent SQL injection" → AI uses parameterized queries but also constructs raw SQL elsewhere

Prevention: - Specify security requirements with explicit test cases - Include security failure scenarios in acceptance criteria - Run security-specific test suites on all AI-generated authentication/authorization code - Have security rules in AGENTS.md that are tested in CI

Failure 5: The Missing Error Path

What happens: Specifications define the happy path in detail but omit error handling. AI implements the happy path correctly but error handling is inconsistent or absent.

Prevention: - For every API endpoint, explicitly specify every error response - Include error scenarios in acceptance criteria - Use a standard error response format specified in AGENTS.md - Test error paths as thoroughly as happy paths

Failure 6: The Semantic Versioning Mismatch

What happens: AI is given context that references library APIs that changed between versions. AI generates code using the API from training data, which doesn't match the installed version.

Examples: - Using findOne from Mongoose v5 when v7 is installed (method removed) - Using Prisma v4 API patterns with v5 installed (breaking changes) - React Query v4 patterns with v5 (renamed hooks)

Prevention: - Always specify exact library versions in AGENTS.md - Include code examples from the specific version used - When using a new major version of a library, add an ADR noting breaking changes

Failure 7: The Autonomy Mismatch

What happens: Agent is given too much autonomy for the task's risk level, or too little for a simple task.

Examples of too much autonomy: - Agent autonomously modifies production database schema - Agent rewrites authentication system without human review - Agent deploys to production without approval gate

Examples of too little: - Human approval required for every line of generated boilerplate - Agent not allowed to run tests to verify its own output - Human must approve formatting and whitespace changes

Prevention: - Define autonomy levels per task type in your workflow - Use isHighRiskAction() gates for irreversible operations - Allow agents to run tests, linters, and formatters autonomously - Require human approval for: production deployments, schema migrations, auth changes, security policy modifications

Chapter 27: The Future of AI Software Engineering

27.1 Where We Are Now

We are at an inflection point in software engineering history. AI coding assistants have moved from curiosity to productivity tool to competitive differentiator in less than three years. The best engineering teams today are experimenting with:

  • AI agents that implement entire features from specifications
  • Multi-agent systems that plan, implement, review, and test autonomously
  • Self-healing systems that detect and fix their own regressions
  • Natural language interfaces to complex codebases

27.2 The Next Five Years

Several trends are converging that will shape the next phase:

Trend 1: Context Windows Keep Growing As context windows grow to millions of tokens, entire codebases will fit in context. This removes many current limitations but increases the importance of well-structured, high-signal specifications.

Trend 2: Persistent Agent Memory AI agents will develop persistent memory across sessions — remembering decisions, learning from corrections, building understanding of your codebase over time. AGENTS.md becomes a seeding mechanism for this memory.

Trend 3: Spec-to-Deployment Pipelines End-to-end pipelines where a specification flows through planning → implementation → testing → review → deployment with minimal human touch for routine features. Humans focus on specifications and review.

Trend 4: Formal Verification Making a Comeback As AI generates more code, formal verification tools that can mathematically prove correctness will become practical. The combination of AI-generated code + formal verification dramatically raises the quality floor.

Trend 5: Specification Languages Evolve New specification languages purpose-built for AI consumption will emerge — richer than markdown, more precise than prose, but more accessible than formal logic. The interface between human intent and AI implementation will become its own engineering discipline.

27.3 What Stays the Same

Despite these changes, several fundamentals will remain:

Specifications will still require human authorship. AI can help draft specifications, but the business decisions, design choices, and domain knowledge that specifications encode require human judgment.

Quality will still be measured against requirements. Tests verify that implementations match specifications. This relationship is fundamental, regardless of who writes the code.

Architecture will still matter. Well-architected systems are easier for AI agents to work with. Poorly architected ones remain difficult regardless of how capable the AI is.

Security will still require explicit specification. AI will not invent security requirements. You must specify them.

27.4 The Developer's Evolving Role

The role of software developers is transforming, not disappearing. The most valuable skills shift from:

Yesterday Tomorrow
Writing syntax correctly Writing specifications precisely
Memorizing API signatures Designing systems thoughtfully
Debugging line by line Understanding system behavior

The developers who will thrive in the next decade are those who develop deep specification skills — the ability to think clearly about what software should do, express it precisely, and verify it rigorously. This is not a diminishment of the craft. It is its elevation.

27.5 Final Thoughts: Specifications as the Human Contribution

Here is the thesis that this guide has been building toward:

As AI becomes more capable at implementing software, the human contribution to software development becomes increasingly about defining what software should do and verifying that it does it.

Specifications are the primary medium of that contribution.

When you write a precise specification, you are doing something AI cannot do: you are translating business reality into technical intent. You are making judgments about what matters, what can be simplified, and what must be precise. You are embedding knowledge about the domain, the users, the constraints, and the context into a form that can guide machines.

That is not a lesser activity than writing code. It is the activity that makes all code worth writing.

Specification-Driven Development is not a restriction on how you work. It is the discipline that ensures your work — and the AI's work on your behalf — actually serves the purpose for which software exists: to solve real problems for real people.

Build with precision. Specify with care. Let AI amplify your judgment, not replace it.
